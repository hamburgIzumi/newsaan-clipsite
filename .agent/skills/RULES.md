## 指示書
*   ユーザーのプロンプトに対し、必ず指示書を作成し各エージェントに作業を渡すこと

##  各エージェント（役割）への指示
*   **タイトル、ISSUEおよびコミットメッセージ、プルリクエストの記述は必ず日本語を使用すること。**
### 役割A: プロジェクトマネージャー (Issue登録・管理)
*   **タスク**:
    1. 本指示書を元に、実装に必要なタスクを細分化する。
    2. 各タスクをGitHubの「Issue」として登録する。
    3. 各Issueには「目的」「実装内容」「完了条件」を明記すること。
*   **Issue構成例**:
    *   `[Issue #1] プロジェクトの基本構造とデザインシステム (HTML/CSS) の構築`
    *   `[Issue #2] Twitch API連携モックまたはデータ構造の定義と読み込み機能`
    *   `[Issue #3] クリップ検索・ソート・日付フィルタ機能 (AND/OR対応) の実装`
    *   `[Issue #4] クリップ再生プレイヤー (サイト内完結) の実装`
    *   `[Issue #5] GitHub Pages 公開設定と最終確認`

### 役割B: フロントエンドエンジニア (開発担当)
*   **タスク**:
    1. 各Issueのタスクを順に消化する。
    2. HTML5/CSS3/JavaScript（モダンVanilla JS）を用いて、要件を満たすコードを記述する。
    3. 機能ごとにコミットを分け、対応するIssue番号をコミットメッセージに含める（例: `feat: クリッププレイヤーの追加 #4`）。
    4. 機能実装が完了したら、GitHubにPushし、プルリクエスト（PR）を作成する。

### 役割C: シニアエンジニア (コードレビュー担当)
*   **タスク**:
    1. フロントエンドエンジニアが作成したプルリクエスト（PR）のコードをレビューする。
    2. **チェックリスト**:
        *   要件（検索・ソート・再生など）がすべて満たされているか？
        *   デザインが「モダンで現代的」かつレスポンシブか？
        *   コードの可読性、変数名、コメント（日本語）は適切か？
        *   不要なコンソールログやデバッグコードが残っていないか？
    3. 必要に応じて修正を指示し、問題がなければマージする。
    4.チェックした内容はPRコメントに残すこと。

## 4. 開発時の補足ルール
*   **Gitブランチ戦略 (厳守)**:
    *   **ベース**: 常に `main` から分岐した **`work` ブランチ** を開発の主軸とします。
    *   **各Issueの開発**: 各Issue用のブランチ（例: `feat/issue-X-...`）は、`main` からではなく、**必ず `work` ブランチから分岐** して作成してください。
    *   **プルリクエスト (PR)**:
        *   各IssueのPRは、`work` ブランチ宛てに作成し、レビュー後に `work` へマージします。
        *   すべてのIssueが消化された後、**`work` ブランチから `main` ブランチへのPR** を作成します。
    *   **マージ制限 (重要)**:
        *   `work` から `main` へのPRを作成した後は、**自動マージ（merge）を行わない** でください（保留状態にし、ユーザーの指示または手動操作に委ねます）。
*   TwitchのAPIからクリップを取得する部分について、APIキーの制限や認証が必要な場合は、静的なJSONデータ（モックデータ）を作成して読み込める設計にし、ローカルでも完全に動作するように配慮すること。
*   コード内のコメントは原則として **日本語** で記述すること。
